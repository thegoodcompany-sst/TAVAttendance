# PDPA Prod Audit — 2026-07-13

Read-only audit of the live prod Supabase project (`zgikcbsxzjgbigywxbbj`) against the PDPA docs,
run before go-live with real student data. Every claim below cites the check that proved it.
No writes were made to the database.

## Verdict

**Technical controls are solid and match the docs.** No broken or missing machinery. The remaining
gaps are the known human/legal items (DPO, sign-off, consent-wording validation) plus one stale
doc reference (HUMANS.md §29). Honest status line unchanged: *technical controls in place, legal
formalisation pending.*

## Advisor findings (security)

`get_advisors(security)` returned **no NEW findings** — every `cache_key` returned is already in
`scripts/advisor-accepted.json`. Namely: the SECURITY DEFINER functions callable by `authenticated`
(all `is_admin()`-guarded / RLS-required, incl. the three PDPA RPCs), `rate_limit_events` RLS-on-no-
policy (service-role by design), and `auth_leaked_password_protection` (WARN). No action beyond
what HUMANS.md already tracks.

- **Leaked-password protection is still OFF** (advisor `auth_leaked_password_protection` = WARN).
  Per HUMANS.md §4 this requires a **paid** Supabase plan and is not actionable on the free tier —
  not a new finding, but flagged in the go-live checklist so it is a conscious decision, not a miss.

## Mismatches found

1. **HUMANS.md §29 is stale.** It says "Decide whether to wire `PdpaPanel` into the web student
   detail page." The panel is already wired and rendering (`page.tsx` line 153). §29 should be
   marked done. (Left as-is per task scope — only added the new checklist section; flag for a
   follow-up edit.)
2. **Consent schema breadth vs. usage.** The `consent_records_consent_type_check` CHECK allows four
   types (`data_collection, result_slips, messaging, photos`) but only `data_collection` is ever
   granted by any UI. This is intentional (admin-attestation model, contract §Consent), not a bug —
   the consent pack therefore uses a single overall consent checkbox rather than four. Noted so the
   form maps 1:1 to what admins can actually record.
3. **Docs still marked DRAFT / DPO placeholder.** Notice v1.1, retention schedule, and breach plan
   all carry "DRAFT — requires DPO sign-off"; the DPO name is a placeholder in the notice and on the
   public page (renders "[email protected]"). Expected — this is the open legal work, not a
   technical defect.
